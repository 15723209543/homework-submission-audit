# 作业提交高预警检查规范

## 检查原则

- 将用户指定的资料文件夹记为 `TASK_DIR`；输入与最终结果必须位于同一文件夹，不能默认使用当前工作目录。
- 将名单作为人员范围标准，将作业要求和模板作为必交内容与格式标准。
- 主动扫描所有有证据支持的问题，不受示例限制。
- 将确定事实与推断风险分开；风险结论写明置信度和复核建议。
- 相同问题复用相同错误码，不同问题使用不同错误码。
- `.errcode` 只登记 `.output` 实际引用的代码。

## 提交者匹配

优先使用完整学号，其次姓名和学号联合证据，最后才使用仅姓名弱匹配。名单学生没有任何可合理归属的提交时返回 `NF`。名单外提交追加在名单记录之后。重名、学号冲突、多人一组、代交或最终版本不明时，不强行归属，登记对应预警。

## 动态错误码

| 前缀 | 分类 | 开放检查范围 |
|---|---|---|
| 1xx | 个人信息错误 | 姓名、学号、班级、归属、文件名与正文身份冲突等 |
| 2xx | 格式错误 | 文件类型、命名、模板、章节、页面、字体、图表、附件位置等 |
| 3xx | 任务完成错误 | 必做题、过程、数据、代码、测试、截图、结果、分析、引用、结论等缺失 |
| 4xx | AI 或抄袭风险 | 多个独立 AI 风险信号、排除公共部分后的高相似内容或代码 |
| 5xx | 文件异常 | 损坏、为空、加密、编码、体积、扩展名伪装、宏、路径、版本等异常 |
| 6xx | 用户额外检查任务错误 | 用户在本次请求中新增的任意检查任务未通过 |

每类在单次任务中从 `x01` 起动态申请。示例不是问题全集；发现新的具体问题时继续分配该类别尚未使用的代码。

## 6xx 扩展接口

调用时先提取用户额外要求，按原意登记到 `extra_checks`。例如：

- 检查是否包含指定声明。
- 检查是否提交源代码、演示视频或附加截图。
- 检查特殊命名、文件夹层级或提交顺序。
- 检查用户指定的统计、关键词、页数、版本或日期条件。

规则：

1. 每项额外检查独立登记，不合并不同要求。
2. 通过时不分配错误码。
3. 未通过时分配 `601`、`602`……并写入对应学生状态和 `.errcode`。
4. 无法检查时标记 `not_checked` 并说明原因，不当作通过；只有用户要求“无法验证也预警”时才分配 6xx。
5. 6xx 的修改建议必须直接对应用户原始要求。

## 高预警扫描

除明确要求外，继续检查：

- 文件和正文身份是否一致。
- 必做项是否遗漏，是否只有标题或占位内容。
- 图片、图表、附件、代码、数据和结论是否可读、可定位、可复现。
- 文档内部是否存在明显矛盾、题目错配、无关内容或异常复制痕迹。
- 文件是否损坏、异常小/大、加密、扩展名不符、重复或最终版本不明。
- 多份提交是否存在排除模板后的罕见共同错误、长段同序文本或非模板代码高度一致。
- 是否出现潜在恶意宏、可执行文件、路径穿越或其他安全异常。

低置信度疑点允许预警，但必须写成“建议人工复核”并保留观察事实。不要把语言流畅、措辞正式或单一检测器分数当作 AI 使用证据。

## 输出规则

`.output` 按名单原顺序输出“姓名、学号、提交情况”，名单外提交最后输出。`SUCC`、`NF` 和错误码互斥，多码用空格分隔。

`.errcode` 输出“错误码分类、错误码、应该如何修改”。4xx 和低置信度项使用风险提示措辞。文件头以 `assets/example.output` 和 `assets/example.errcode` 为准，文件名行替换为实际名称。两个文件使用严格 GBK，不用替换字符掩盖编码错误。

将两个最终文件直接写到 `TASK_DIR` 根目录。中间 JSON、清单、解压内容和证据只放在 `<TASK_DIR>/.hfut-homework-audit/`。无论 Codex 当前工作目录是什么，都不得把最终结果写到当前目录或 Skill 安装目录。

## 降级规则

- 缺少模板：不判断模板一致性。
- 缺少名单：不能可靠生成完整 `NF`。
- 缺少要求：只检查提交存在性和可直接确认的异常。
- 不支持或无法打开的压缩包：记录未解析范围，不能把包内学生判为 `NF`。
- 未知代码默认静态检查，不直接执行。
